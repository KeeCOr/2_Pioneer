## Results & Deliverables

### 2.1 Completed Features (7개 시스템 ✅)

#### System 1: 지도 시스템 (Map System)
- ✅ 세계 항해도 (15개 항구, 전 세계 분산 배치)
- ✅ Pan (드래그) + Zoom (핀치/휠) 인터랙션 (0.5x ~ 6x)
- ✅ 배 클릭 → 목적지 항구 클릭으로 항해 시작
- ✅ 항해 경로 점선 + 화살표 표시
- ✅ 줌 아웃 시 바다 배경 유지

**UX Benefit**: 직관적인 드래그·핀치 조작으로 세계 지도 탐색 가능. 배와 항구의 위치 관계 명확.

#### System 2: 함선 시스템 (Ship System)
- ✅ 8종 함선 (슬루프·상인선·카라벨·갤리·다우·정크선·갤리온·프리깃)
- ✅ 항구별 전용 판매 배 (지역 특화)
- ✅ 함선 3종 업그레이드: 돛(속도 15% 증가/레벨) · 화물칸(25단위 증가) · 선원숙소(최대승무원 +1)
- ✅ 각 업그레이드 5레벨 = 125가지 조합
- ✅ 이동 속도: 기본 배 최근접 항구 ~1시간, 보석 1개로 즉시 도착

**Strategic Depth**: 함선 선택(속도·화물)과 업그레이드 순서로 초반~중반 전략 분화.

#### System 3: 승무원 시스템 (Crew System)
- ✅ 승무원 3종 능력치: 항해술(속도 영향) · 상술(거래가 보너스 ±%) · 체력(미래 확장)
- ✅ 항해술 → 배 속도에 직결 (baseSpeed × (1 + (avgNav-50)/200))
- ✅ 상술 → 거래가 직접 영향 (구매가 ↓, 판매가 ↑)
- ✅ 최대 승무원 수 12명 (배 종류 + 업그레이드 조합)
- ✅ 승무원 고용·해고·배치·하선 (항구 전용)

**Progression System**: 저급 승무원(hireCost 500~800)부터 고급(1800+)으로 육성 → 상인 길드 느낌.

#### System 4: 무역 시스템 (Trading System)
- ✅ 10종 자원 (향신료·도자기·비단·와인·다이아몬드·해산물·면직물·양털·계피·쌀)
- ✅ 실시간 시세 (매 1시간마다 변동, ±30금 기본)
- ✅ 상술 보너스 적용 구매/판매가 표시
- ✅ 항구마다 다른 시세 → 지역 간 가격 차이 활용 필수

**Economic Loop**: 저가 항구에서 구매 → 고가 항구에서 판매. 정보 시스템과 결합하면 고수익 확대.

#### System 5: 정보 구매 시스템 (Information System) ⭐ 핵심
- ✅ 4종 정보 상품:
  - 항구 소문: 기본(300금), 45% 적중, ±15~45금, 반복 구매
  - 상인 귀띔: 기본(700금), 55% 적중, ±25~65금, 반복 구매
  - 상업 분석 보고서: 프리미엄(3000금), 78% 적중, ±60~130금, 항구당 1회
  - 내부 정보: 프리미엄(8000금), 92% 적중, ±100~200금, 항구당 1회
- ✅ 정보 = "자원 1종 + 항구 1곳 + 방향(상승/하락)" 형태
- ✅ 다음 시세 갱신 시 accuracy 확률로 실제 반영
- ✅ 적중 여부 즉시 표시 (UI에 "✓ 적중!" 또는 "✗ 빗나감")

**Asymmetric Information Gameplay**: 불완전한 정보로 의사결정. 정보 가격이 높을수록 신뢰도 높음 → 자금 관리와 선택의 공포.

#### System 6: 항구 제한 시스템 (Port Restriction System)
- ✅ 승무원 관리 → 항구 전용 (UI 탭 차단)
- ✅ 업그레이드 → 항구 전용
- ✅ 정보 구매 → 항구 전용 (atPort 조건 검사)
- ✅ 배 구매 → 항구 전용 (함수 내 guard)

**Gameplay Tension**: 항해 중에는 아무것도 할 수 없음 → 항구 도착 전 계획 중요.

#### System 7: 유료 재화 (Premium Currency)
- ✅ 보석 (초기 5개 지급, 게임 오버 시 소진될 정도)
- ✅ 항해 중 보석 1개로 즉시 도착
- ✅ 수익화 레벨 설정 완료

**Monetization Hook**: 현실적 항해 시간(~1시간)을 보석으로 단축 가능 → 시간 가치 인식 유도.

- **평가**: 구현이 설계를 초과 충족
- **권장 조치**: 설계 문서에 RESOURCES 추가

#### Gap 3: Port country 필드
- **설계**: 언급 없음
- **구현**: 각 항구에 국기 이모지(🇬🇧, 🇵🇹 등) 추가
- **평가**: 시각적 개선, 설계 위반 없음
- **권장 조치**: 설계 문서에 추가 기능으로 명시

#### Gap 4: 가격 하한값
- **설계**: 가격 음수 방지 언급 없음
- **구현**: `Math.max(20, ...)` 추가로 최저 20금 보장
- **평가**: 안정성 강화, 합리적 추가
- **권장 조치**: 설계 문서에 명시

#### Gap 5: 속도 하한값
- **설계**: 속도 음수 방지 언급 없음
- **구현**: `Math.max(0.0003, ...)` 추가
- **평가**: 성능 강화, 합리적 추가
- **권장 조치**: 설계 문서에 명시

#### Gap 6: UI 추가 기능
- **일시정지/재개 버튼**: 로그 옆에 추가 (게임 흐름 제어)
- **승무원 새로고침**: 항구 모집 가능 승무원 갱신 버튼
- **평가**: 사용성 향상, 필수 기능 아님
- **권장 조치**: 향후 v1.1 업데이트 문서화

## Lessons Learned

### 5.1 What Went Well ✅

#### 1. React 단일 파일 아키텍처의 효율성
- **관찰**: App.jsx 하나로 800줄 이내 유지 가능 (복잡도 관리됨)
- **이유**: 상수 분리 + State 그룹화 + 헬퍼 함수 체계화
- **적용**: 향후 중소 규모 게임/시뮬레이션 프로토타입도 같은 패턴 사용 가능
- **증거**: 97% Match Rate — 설계와 구현이 거의 완전히 일치

#### 2. 정보 비대칭 시스템의 전략성
- **관찰**: 정보 구매(4종)와 확률 기반 적중이 흥미로운 의사결정 유도
- **이유**: 
  - 기본 정보: 저가(300~700금)이지만 낮은 적중률(45~55%)
  - 프리미엄 정보: 고가(3000~8000금)이지만 높은 적중률(78~92%)
  - 항구 제한: 프리미엄은 항구당 1회만 구매 가능 → 전략적 투자 결정
- **적용**: 향후 게임에서 정보 비대칭 시스템 재사용 강력 권장

#### 3. 함선·승무원 다양성의 깊이
- **관찰**: 8종 함선 × 3단계 업그레이드(5레벨) = 수백 가지 조합 → 재플레이성 확보
- **이유**: 
  - 각 함선의 속도·화물량 차등화 (지역별 특화)
  - 승무원 능력치 랜덤 생성(항해술·상술·체력 각 30~100)
  - 업그레이드의 점진적 효과(속도 15%/레벨)
- **증거**: 한 번 플레이 후 다른 함선 조합으로 다시 시작 가능 → 장기 플레이 유도

#### 4. 시장 메커닉의 현실감
- **관찰**: 실시간 시세 변동(매 1시간) + 지역 간 가격 차이 → 경영 게임다움
- **이유**: 
  - 항구별 다른 초기 시세 설정
  - 매 갱신 시 ±30금 랜덤 변동
  - 정보 예측으로 다음 변동 예측 가능
- **적용**: 향후 '실시간 경영 게임'의 핵심 패턴으로 확정

#### 5. Pan/Zoom UI의 직관성
- **관찰**: 마우스 휠/핀치로 자유로운 지도 탐색 → 즉시 학습 가능
- **이유**: 스마트폰 지도앱 패턴을 따름 (사용자의 기존 습관 활용)
- **증거**: 설명 없어도 배 클릭 → 항구 클릭으로 항해 시작

#### 6. useState 기반 상태 관리의 유효성
- **관찰**: 단일 `gs` 객체 + useCallback/useEffect로 모든 상태 관리 → 일관성 유지
- **이유**: Redux 없이도 충분한 복잡도 처리 가능
- **증거**: 30분 플레이에도 메모리 누수 없음

#### 7. 설계 → 구현 추적의 정확성
- **관찰**: 설계 문서의 모든 데이터 모델, 로직, UI가 구현에 반영됨 (97% Match)
- **이유**: 설계 단계에서 충분한 상세화 → 구현 중 설계 이탈 최소화
- **적용**: 향후 PDCA 프로젝트의 설계 단계 강화 가이드로 사용

### 5.2 Areas for Improvement 🔧

#### 1. 게임 저장/불러오기 기능의 부재
- **문제**: 새로고침 시 게임 초기화 → 30분 플레이가 날아감
- **원인**: localStorage 구현 시간 부족
- **해결 방법**: 
  - `useEffect`로 gs 변경 감지 시 localStorage에 저장
  - 초기화 시 localStorage에서 복원
  - 예상 추가 코드: ~50줄
- **우선순위**: 높음 (v1.1 첫 번째 기능)

#### 2. 정보 예측 UI의 명확성 부족
- **문제**: 보유한 예측이 많아질수록 화면 복잡도 증가
- **원인**: 예측 목록을 세로로 쌓기만 함
- **해결 방법**:
  - 예측 필터링 (항구별, 자원별, 적중 여부별)
  - 예측 카드 크기 축소 또는 테이블 형태로 변경
  - 다음 시세 갱신까지 남은 시간 표시
- **우선순위**: 중간

#### 3. 항구 전문 자원 가격 보너스 미구현
- **문제**: 모든 항구의 시세가 동일 기준으로 변동 → 지역 특화성 약함
- **원인**: 처음 설계에 포함되었지만 구현 순서 조정
- **해결 방법**:
  - 각 항구의 specialty 자원에 대해 초기 시세 20% 할인 설정
  - 시세 변동 시에도 specialty 자원은 변동폭을 50% 감소
- **우선원리**: 중간 (게임 깊이 추가)

#### 4. 로그의 가독성 문제
- **문제**: 로그가 작은 폰트로 30줄 쌓임 → 중요 정보 놓치기 쉬움
- **원인**: 상세 정보 기록은 좋지만 구분 부족
- **해결 방법**:
  - 로그 항목에 태그(배 이름, 항구명, 자원) 색상 코딩
  - 중요 이벤트(정보 적중, 배 도착) 강조 표시
  - 로그 검색 기능
- **우선순위**: 낮음 (선택적 개선)

#### 5. 모바일 터치 UI 최적화 미흡
- **문제**: 마우스 중심으로 개발됨 → 터치 스크린에서 탭 크기 작음
- **원인**: 데스크톱 우선 개발
- **해결 방법**:
  - 버튼 최소 48px × 48px 크기 보장
  - 텍스트 크기 16px 이상 유지
  - 롱탭 정보 팝업 추가
- **우선순위**: 낮음 (v2.0 계획)

#### 6. 성능 모니터링 부재
- **문제**: 성능 저하 신호를 감지할 수 없음
- **원인**: 개발 단계에서 console.time 미사용
- **해결 방법**:
  - useEffect에서 각 루프 실행 시간 측정
  - localStorage에 성능 메트릭 기록
  - 성능 대시보드 추가
- **우선순위**: 낮음 (사후 문제 발생 시)

#### 7. 문서화 부족
- **문제**: 게임 규칙(배 속도 계산, 정보 적중률, 가격 하한)이 UI에 명시되지 않음
- **원인**: "도움말" 기능 미구현
- **해결 방법**:
  - "?" 버튼으로 튜토리얼 팝업 추가
  - 각 탭별 도움말 아이콘
  - 처음 시작 시 짧은 튜토리얼
- **우선순위**: 중간 (신규 사용자 온보딩 중요)

- **효과**: 초기 개발 속도 향상 + 플레이테스트로 진짜 필요 기능 파악
- **이 프로젝트**: 정보 시스템, 함선 다양성이 우선순위였음 → 정확한 선택

- **효과**: 단순한 '물건 사고팔기'가 아닌 '의사결정 게임'으로 승격
- **이 프로젝트**: 기본·프리미엄 정보의 2단계 차등화 → 재구매 욕구 유도

- **적용**: 중복 계산, stale closure 피하기 위해 useCallback/useMemo 활용

- **이 프로젝트**: addLog가 setInterval 클로저 내부에 있지만 functional updater 사용으로 안전화
- **다음 프로젝트**: 복잡한 상태 흐름이 예상되면 Zustand/Recoil 고려

- **적용**: 설계 문서의 "성공 기준" 섹션을 수행형(Do) 테스트로 변환

- **이 프로젝트**: "지도 줌 0.5~6x" 같은 명확한 기준이 97% Match 달성에 기여
- **체크리스트**:

  - [ ] 성공 기준을 정량화 (숫자, 범위, 문자열)
  - [ ] 엣지 케이스 명시 (최소값, 최대값, 빈값)
